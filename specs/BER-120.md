# BER-120: Kontenrahmen mandantenfähig machen (+ SKR03-Reste bereinigen)

> **Pre-Flight:** Lies `CLAUDE.md`, dann `docs/UEBERGABE.md`, dann
> `docs/audits/2026-07-23-strukturpruefung-ausbaustufe.md`.
> **Einordnung:** Diese Story wird in dieser Stufe SPEZIFIZIERT, aber NICHT im
> Baulauf gebaut — sie ändert den Konten-Primärschlüssel und erfordert einen
> n8n-Live-Patch (Betreiber-Schritt am Tag). Keine Abhängigkeit von BER-116..119
> rückwärts; Voraussetzung für die AUSLIEFERUNG des Korrekturstapels ist nur die
> Umbuchungsliste der 6 fehlkontierte 6520-Belege im K2-Anschreiben (erledigt,
> s. u.), nicht diese Story.

## Kontext (verifiziert gegen Prod, 23.07.2026)

1. **`skr04_konten` ist global** — keine `firma_nr`, alle Kunden teilen 35 Zeilen.
   Im Abo-Modell der Blocker: kein SKR03-Kunde bedienbar, keine individuellen
   Kontenlisten. `kontierungs_lerndata` trägt bereits `mandant_id`; der Rahmen
   darunter nicht.
2. **Die Tabelle mischt Kontenrahmen:** 1000 Kasse, 1200 Bank, 1571/1575/1576
   Vorsteuer sind SKR03 (SKR04: 1600, 1800, 1401/1405/1406); 3801/3806, 4300/4400
   sind SKR04. Der Export-Header meldet SKR `04`. Zusätzlich fachlich fraglich im
   SKR04: 6520 „Gewerbesteuer" (Standard 7610; 65xx = Fahrzeugkosten) mit **6
   festgeschriebenen Belegen**, 6300 „Löhne und Gehälter" (Standard 60xx), und ein
   Duplikat „Bewirtungskosten" auf 6870 (kanonisch 6640; 0 Belege auf 6870).
3. **n8n validiert KI-Vorschläge gegen eine hartkodierte Kontenliste** in vier
   Code-Nodes (Threema- und PDF-Workflow, Kommentar „bei neuen Konten hier
   nachziehen") — jede Kontenänderung braucht den Live-Patch + Repo-Export.
4. FKs auf `skr04_konten(konto_nr)`: `belege.sachkonto`,
   `kontierungs_lerndata.sachkonto_vorschlag/final` (Tabelle leer).

## Anforderung

Mehrere Kontenrahmen nebeneinander, Zuordnung pro Firma; vorhandene Zeilen konsistent
zum Rahmen der jeweiligen Firma; fehlende Konten ergänzt; Umgang mit den 6
festgeschriebenen 6520-Belegen geklärt (nur Klärung — `sachkonto` bleibt eingefroren,
BER-119 lockert es bewusst nicht).

## Entschiedenes Design

### Datenmodell

Physischer Tabellenname bleibt `skr04_konten` (vier n8n-Code-Nodes, App-Queries und
FKs hängen daran; ein Rename hätte nur kosmetischen Wert). Die Mandantenfähigkeit
kommt über Spalten; ein `COMMENT ON TABLE` stellt klar: „Kontenstamm aller Rahmen —
Name historisch."

```sql
-- Migration BER-120 (Entwurf — bei Umsetzung nach threema-decrypt kopieren):
CREATE TABLE public.kontenrahmen (
  code        text PRIMARY KEY,              -- 'SKR03' | 'SKR04' | künftig kanzleiindividuell
  bezeichnung text NOT NULL,
  created_at  timestamptz NOT NULL DEFAULT now()
);
INSERT INTO public.kontenrahmen (code, bezeichnung) VALUES
  ('SKR04', 'DATEV Standardkontenrahmen 04'),
  ('SKR03', 'DATEV Standardkontenrahmen 03');

ALTER TABLE public.firmen
  ADD COLUMN kontenrahmen_code text NOT NULL DEFAULT 'SKR04'
    REFERENCES public.kontenrahmen(code);

ALTER TABLE public.skr04_konten
  ADD COLUMN kontenrahmen_code text NOT NULL DEFAULT 'SKR04'
    REFERENCES public.kontenrahmen(code),
  ADD COLUMN firma_nr char(2) REFERENCES public.firmen(firma_nr);
    -- NULL = Standardzeile des Rahmens; gesetzt = firmenindividuelle Zeile/Override

-- PK-Umbau: (konto_nr) → Surrogat + Eindeutigkeit je Rahmen/Firma.
-- Reihenfolge zwingend: erst abhängige FKs droppen, dann PK.
ALTER TABLE public.belege DROP CONSTRAINT belege_sachkonto_fkey;
ALTER TABLE public.kontierungs_lerndata
  DROP CONSTRAINT kontierungs_lerndata_sachkonto_vorschlag_fkey,
  DROP CONSTRAINT kontierungs_lerndata_sachkonto_final_fkey;
ALTER TABLE public.skr04_konten DROP CONSTRAINT skr04_konten_pkey;
ALTER TABLE public.skr04_konten ADD COLUMN id uuid NOT NULL DEFAULT gen_random_uuid();
ALTER TABLE public.skr04_konten ADD PRIMARY KEY (id);
CREATE UNIQUE INDEX idx_konten_rahmen_firma_nr
  ON public.skr04_konten (kontenrahmen_code, konto_nr, COALESCE(firma_nr, '--'));
```

**Integritätsersatz für die gedroppten FKs** (ein Komposit-FK würde
`kontenrahmen_code` an `belege` erzwingen — abgelehnt, der Rahmen hängt an der
Firma): Trigger `fn_belege_sachkonto_gueltig` BEFORE INSERT OR UPDATE OF sachkonto ON
belege — prüft `EXISTS (Zeile im Rahmen der Firma des Mandanten: firmenindividuelle
Zeile vor Standardzeile, `ist_aktiv IS NOT false` nur bei INSERT/Statuswechsel nach
vorn, nicht rückwirkend)`. Fehlertext:
`'Sachkonto % existiert nicht im Kontenrahmen % der Firma %'`.

Auflösungsregel überall (App + Trigger): erst `firma_nr = <firma>`, dann
`firma_nr IS NULL` im `kontenrahmen_code` der Firma.

### Datenbereinigung (Seeds derselben Migration)

| Aktion | Zeilen | Begründung |
|---|---|---|
| `kontenrahmen_code='SKR03'` setzen + `ist_aktiv=false` | 1000, 1200, 1571, 1575, 1576 | SKR03-Nummern; keine Belege referenzieren sie; als SKR03-Zeilen bleiben sie für künftige SKR03-Firmen korrekt benannt |
| SKR04-Pendants einfügen (aktiv) | 1600 Kasse, 1800 Bank, 1810 Bank 2 (Alternativkonto BER-116), 1401 Abziehbare Vorsteuer 7 %, 1406 Abziehbare Vorsteuer 19 %, 2100 Privatentnahmen allgemein, 2180 Privateinlagen, 7610 Gewerbesteuer | fehlende SKR04-Standardkonten inkl. der drei Gegenkonten aus BER-116 |
| `ist_aktiv=false` | 6870 (Bewirtungs-Duplikat), 6520 (Betreiber ist Freiberufler — GewSt fällt nicht an; das Konto hat der KI nachweislich Fehlkontierungen ermöglicht, 23.07.2026) | keine neuen Fehlkontierungen; bestehende 6 Belege bleiben unberührt (Anzeige liest Rohwert, `ist_aktiv` filtert nur Auswahl/KI) |
| 7610 Gewerbesteuer | für **gewerbliche** Betatest-Kunden (`docs/FEATURE-WUENSCHE.md` F-03); für die Betreiber-Firma ohne Belegung | Produkt-Vorbereitung, kein Betreiber-Bedarf |
| 6300 bleibt aktiv | — | Lohnkonten erst bei Bedarf (F-04); aktuell keine Löhne (Betreiber 23.07.2026) |

(Sollte je ein Konto doch gebraucht werden: `ist_aktiv=true` per Konfig-SQL
zurück — kein Code.)

### App-Anpassungen

- Alle Konten-Reads firmenscopen: Detailseite (`konten`-Select), Freigabe-Validierung,
  BER-118-Anlage-Validierung, Nacherfassungs-Query (Join bleibt, zusätzlich Rahmen der
  Session-Firma), `vorsteuer_relevant`-Gate — gemeinsame Helper-Funktion
  `kontenFuerFirma(firmaNr)` in neuem `src/lib/konten.ts`.
- `skr04_lesen`-RLS-Policy (public/authenticated) bleibt; `dash_skr04_select` bleibt
  `USING (true)` — die Firmen-Scopung ist Anwendungslogik (Konten sind keine
  Geheimnisse zwischen Mandanten desselben Betreibers; personenbezogen ist daran
  nichts). Bewusst KEINE RLS-Verschärfung in dieser Story.
- `docs/DATEV.md`: Header-SKR aus `firmen.kontenrahmen_code` ableiten
  (`'"04"'` → `q(rahmen.slice(-2))`).

### n8n (Betreiber-Schritt, dokumentiert im Runbook der Umsetzung)

Die vier Code-Nodes: hartkodierte `SKR04_KONTEN`-Liste um die neuen aktiven Konten
ergänzen bzw. — bevorzugt, wenn ohnehin angefasst — durch einen Supabase-REST-Read
`skr04_konten?ist_aktiv=eq.true&kontenrahmen_code=eq.SKR04&firma_nr=is.null&select=konto_nr`
ersetzen (Cache im Workflow-Static-Data, TTL 1 h). Live-Patch per API + Repo-Export
nachziehen (Arbeitskonvention).

### Die 6 festgeschriebenen 6520-Belege — geklärt (23.07.2026)

Befund: keine Gewerbesteuer (Betreiber ist Freiberufler), sondern **Fehlkontierungen
der KI** (5× Steuerberater-Honorar → 6830, 1× Werbung → 6880; „Steuer…" im
Verwendungszweck hat 6520 getriggert). `sachkonto` bleibt eingefroren — die
Umbuchung übernimmt die Kanzlei anhand der Liste im K2-Anschreiben
(`docs/OFFENE-FRAGEN-STB.md`, Versandteil Punkt 2). Lernpunkt fürs
Kontierungsgedächtnis (BER-98) und Beleg für die 6520-Deaktivierung oben.

## Akzeptanzkriterien (prüfbar)

- [ ] `kontenrahmen` existiert mit SKR03/SKR04; `firmen.kontenrahmen_code` = 'SKR04'
      für 01 und 99.
- [ ] `skr04_konten`: Surrogat-PK; UNIQUE (rahmen, konto_nr, firma) via Index;
      die fünf SKR03-Zeilen tragen `SKR03` + inaktiv; die acht SKR04-Ergänzungen
      existieren aktiv; 6870 und 6520 inaktiv, 6520-Zeile unverändert vorhanden.
- [ ] Die 6 Belege auf 6520 sind unverändert (SELECT belegt: `sachkonto='6520'`,
      Status `exportiert`) und die Detailansicht zeigt sie weiterhin an.
- [ ] FK-Ersatz wirkt: INSERT eines Belegs mit `sachkonto='9999'` schlägt mit dem
      definierten Fehlertext fehl; mit `sachkonto='1600'` gelingt er
      (Rollback-Transaktions-Test).
- [ ] Freigabe-Konto-Auswahl zeigt nur aktive Konten des Firmen-Rahmens
      (1600/1800/… sichtbar, 1000/1200/6870/6520 nicht).
- [ ] Kontierungs-KI: n8n-Liste aktualisiert (Live + Repo-Export identisch —
      Betreiber bestätigt im PR-Kommentar; ohne diese Bestätigung bleibt die Story
      offen).
- [ ] `docs/SCHEMA.md` + `docs/DATEV.md` (SKR-Header) nachgezogen.
