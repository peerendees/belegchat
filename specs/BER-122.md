# BER-122: Mehrere MwSt-Sätze pro Beleg

> **Pre-Flight:** Lies `CLAUDE.md`, dann `docs/UEBERGABE.md`, dann
> `docs/audits/2026-07-23-strukturpruefung-ausbaustufe.md`.
> **Einordnung:** Nächstes Release. In dieser Stufe wird NUR sichergestellt, dass
> BER-117 die Erweiterung nicht ausschließt (erledigt — s. „Additivitäts-Nachweis").
> Diese Spec ist die baureife Grundlage für die spätere Umsetzung.

## Kontext

Restaurant- und Supermarktbelege führen 7 % und 19 % nebeneinander. Heute trägt
`belege` genau einen Satz (`mwst_satz`, `mwst_betrag`, `betrag_netto`), BER-108
leitet Teilbeträge aus dem einen Satz ab, der Export schreibt eine Zeile pro Beleg,
und BER-117 setzt genau einen `bu_schluessel` pro Beleg. Die n8n-Plausibilität
(BER-110) prüft bewusst nur `netto + mwst = brutto` und verträgt Mischsätze bereits.

## Anforderung

Ein Beleg kann n Steuerzeilen tragen (Satz, Netto, MwSt, Schlüssel je Zeile); der
DATEV-Export schreibt dann n Buchungssätze. Die Erweiterung ist additiv: kein
festgeschriebener Beleg wird verändert, Ein-Satz-Belege funktionieren unverändert.

## Entschiedenes Design

### Datenmodell (Migration dieser Story — Entwurf, bei Umsetzung ins Schwesterrepo)

```sql
CREATE TABLE public.beleg_steuerzeilen (
  id            uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  beleg_id      uuid NOT NULL REFERENCES public.belege(id) ON DELETE CASCADE,
  pos           smallint NOT NULL CHECK (pos > 0),
  mwst_satz     numeric(5,2) NOT NULL,
  betrag_netto  numeric(12,2) NOT NULL,
  mwst_betrag   numeric(12,2) NOT NULL,
  betrag_brutto numeric(12,2) GENERATED ALWAYS AS (betrag_netto + mwst_betrag) STORED,
  bu_schluessel varchar(4) CHECK (bu_schluessel IS NULL OR bu_schluessel ~ '^[0-9]{1,4}$'),
  UNIQUE (beleg_id, pos)
);
```

Semantik:
- **0 Zeilen** = Ein-Satz-Beleg wie heute (`belege.mwst_satz`/`bu_schluessel` gelten).
  Der GESAMTE Bestand und alle künftigen Ein-Satz-Belege bleiben in diesem Modus —
  es findet KEINE Rückmigration statt.
- **≥ 2 Zeilen** = Mehrsatz-Beleg: die Zeilen sind die Wahrheit; `belege.mwst_satz`
  und `belege.bu_schluessel` sind dann NULL (Konsistenz-Trigger erzwingt:
  Steuerzeilen und Beleg-Einzelsatz schließen sich aus; Summe der Zeilen ==
  `betrag_netto`/`mwst_betrag`/`betrag_brutto` des Belegs auf den Cent).
  Genau 1 Zeile ist verboten (dann Ein-Satz-Modus verwenden) — Konsistenz-Trigger.
- Festschreibung: eigener Trigger auf `beleg_steuerzeilen` nach dem Muster
  `fn_beleg_seiten_unveraenderbar` — INSERT/UPDATE/DELETE nur solange der Beleg
  NICHT festgeschrieben ist. (Die `belege`-Whitelist bleibt unberührt — der
  Satellit hat seinen eigenen Schutz; genau dieses Muster hat die Whitelist-
  Entscheidung aus der Strukturprüfung §2.1 vorgesehen.)

### Erfassung

- Eingang (n8n/OCR): KI-Schema um optionales Array `steuerzeilen[{satz, netto,
  mwst}]` erweitern; liefert die OCR nur Summen, entsteht ein Ein-Satz-Beleg wie
  heute (kein Zwang zur Zeilenbildung). n8n-INSERT der Zeilen via Service Role.
- Freigabe-Formular: Abschnitt „Steuerzeilen" (nur sichtbar, wenn Zeilen existieren
  oder der Nutzer „Beleg hat mehrere Steuersätze" aktiviert): editierbare Zeilen
  (Satz aus {0,5,7,16,19}, Netto, MwSt), Live-Summenprüfung gegen den
  Dokumentbetrag, Schlüssel je Zeile vorbelegt nach BER-117-Gate
  (`vorsteuer_relevant` des Sachkontos + `steuerschluessel`-Mapping je Zeilensatz).
- Freigabe-Route: validiert Summen (±0,01 € Toleranz wie BER-110-Muster: exakte
  Centgleichheit gefordert, Fehler 422 `"Steuerzeilen summieren nicht auf den
  Belegbetrag"`), persistiert Zeilen VOR dem Status-UPDATE.

### Wechselwirkungen (verbindlich entschieden)

- **BER-108 Teilbetrag:** Teilbetrag und Mehrsatz schließen sich in v2 aus —
  Freigabe mit beidem → 422 `"Teilbetrag bei mehreren Steuersätzen noch nicht
  unterstützt"`. (Ein Teil-Split über mehrere Sätze braucht eine Zuordnung, welcher
  Satzanteil gebucht wird — eigene Folge-Story, Feld `gebucht`-Pendant je Zeile ist
  im Modell vorgesehen, aber nicht gebaut.)
- **BER-118:** Manuelle Erfassung bleibt Ein-Satz (Formular ohne Zeilen-Editor in
  v2-Erstausbau; Zeilen können vor der Freigabe im Freigabe-Formular ergänzt werden).
- **DATEV-Export:** `belegRow(b, meta)` → `belegRows(b, meta): string[]` — bei
  0 Steuerzeilen exakt die heutige eine Zeile; bei n Zeilen n Zeilen mit: Umsatz =
  Zeilen-Brutto, gleiches Konto/Gegenkonto/Belegdatum, **Belegfeld 1 identisch**
  (`beleg_nr` — DATEV gruppiert die Sätze über Belegfeld 1 zum Beleg),
  BU-Schlüssel der Zeile in Spalte 9, Buchungstext + Suffix ` (<satz> %)`.
  `anzahl_belege` in `datev_exporte` zählt weiterhin Belege; neue Spalte wird NICHT
  eingeführt (Zeilenzahl ergibt sich aus der Datei).
- **BER-121:** unverändert gültig — Fassungen speichern Bytes, egal wie viele
  Zeilen ein Beleg erzeugt.

### Additivitäts-Nachweis (warum BER-117 nichts zementiert)

Kein festgeschriebener Beleg muss angefasst werden: Bestandsbelege bleiben im
Ein-Satz-Modus dauerhaft gültig; der Mehrsatz-Modus entsteht ausschließlich für NEUE,
noch offene Belege über eine Satellitentabelle mit eigenem Festschreibungs-Trigger;
der Export wechselt auf Code-Ebene von „eine Zeile pro Beleg" zu „Zeilen pro Beleg".
Einzige Vorleistungen in BER-117 (bereits umgesetzt): Schlüssel-Zuordnung liegt als
Konfigurationstabelle mit `mwst_satz`-Schlüssel vor (je Zeile wiederverwendbar), und
die Whitelist-Trigger-Architektur lässt Satellitentabellen zu, ohne die
`belege`-Sperrliste zu ändern.

## Akzeptanzkriterien (prüfbar, für die spätere Umsetzung)

- [ ] Tabelle + Konsistenz-Trigger: 1 Zeile → Fehler; 2 Zeilen mit Summen ≠
      Belegbeträge → Fehler; 2 korrekte Zeilen + `belege.mwst_satz IS NOT NULL` →
      Fehler; korrekt (Zeilen + NULL-Einzelsatz) → ok. Jeweils
      Rollback-Transaktions-Test.
- [ ] Zeilen eines festgeschriebenen Belegs sind unveränderlich (INSERT/UPDATE/
      DELETE schlagen fehl — Testtrio wie `beleg_seiten`).
- [ ] Freigabe eines 7/19-Restaurant-Testbelegs erzeugt zwei Steuerzeilen; Export
      enthält zwei Datenzeilen mit gleichem Belegfeld 1, korrekten Beträgen und
      Schlüsseln (`80`/`90` gemäß `steuerschluessel`-Konfiguration); Summe der
      Umsätze == Dokumentbrutto.
- [ ] Ein-Satz-Beleg erzeugt bit-identische Exportzeile wie vor der Story
      (Regressionstest gegen konservierte Fixture).
- [ ] Teilbetrag + Steuerzeilen → definierter 422.
- [ ] `docs/DATEV.md` (Buchungslogik: „eine Zeile pro Steuerzeile"), `docs/SCHEMA.md`,
      `docs/UEBERGABE.md` (v1-Beschränkung entfernen) nachgezogen.
