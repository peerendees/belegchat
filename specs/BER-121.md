# BER-121: DATEV-Exporte nachvollziehbar — Inhalt, Hash, Versionierung

> **Pre-Flight:** Lies `CLAUDE.md`, dann `docs/UEBERGABE.md`, dann
> `docs/audits/2026-07-23-strukturpruefung-ausbaustufe.md`.
> **Voraussetzungen:** Konsolidierte Migration (S1); BER-116/117/118 gemerged
> (`datev.ts`/Export-Routen werden hier weiterbearbeitet). **Blockiert BER-119**
> (dessen Commit ruft `friereExportEin` auf). Teil des Baulaufs (Schritt S4).

## Kontext

`datev_exporte` speichert weder Dateiinhalt noch Hash; der Re-Download regeneriert aus
den Belegen und liest Firmen-Konfiguration zur Downloadzeit — eine spätere Änderung
veränderte ausgelieferte Stapel rückwirkend und unbemerkt (verifiziert; die
gegenteilige Aussage in `docs/DATEV.md` Z. 29–31 ist falsch und wird korrigiert).
Mit BER-119 wird aus dem theoretischen ein akutes Problem: Nacherfasste Belege würden
die Regeneration der bereits ausgelieferten Datei verändern. BelegChat könnte nicht
mehr belegen, was ursprünglich ausgeliefert wurde.

## Anforderung

1. Jeder Export hält fest, welcher Dateiinhalt ausgeliefert wurde (Bytes + SHA-256).
2. Eine Korrekturfassung ist eine neue, als solche erkennbare Export-Zeile mit
   Referenz auf die ersetzte Fassung — keine stille Ersetzung.
3. Ersetzungs- und Einfrier-Ereignisse stehen mit Anlass im Audit-Trail.

## Entschiedenes Design

### Datenmodell (bereits in der konsolidierten Migration — nicht neu anlegen)

| Spalte auf `datev_exporte` | Typ | Bedeutung |
|---|---|---|
| `version` | `smallint NOT NULL DEFAULT 1` | Fassungszähler je Stapel-Wurzel |
| `wurzel_export_id` | `uuid NULL REFERENCES datev_exporte(id)` | NULL = selbst Wurzel; Korrekturen zeigen auf die Version-1-Zeile. Code liest `COALESCE(wurzel_export_id, id)` |
| `ersetzt_export_id` | `uuid NULL REFERENCES datev_exporte(id)`, UNIQUE-Index (partiell) | die unmittelbar ersetzte Fassung; jede Fassung ist höchstens einmal ersetzbar |
| `korrektur_grund` | `text NULL` | Pflicht bei Korrekturfassungen |
| `inhalts_hash` | `text NULL`, CHECK `^[0-9a-f]{64}$` | SHA-256 über die Dateibytes |
| `datei_inhalt` | `bytea NULL` | die Datei selbst (EXTF ≈ 30 KB — bewusst in der DB: `dashboard_service` hat keinen Storage-Zugriff, und der Inhalt gehört transaktional zum Datensatz; generischer Name, derselbe Mechanismus trägt später den Vollständigkeitsexport BER-123) |
| `eingefroren_am` | `timestamptz NULL` | ab hier sind Inhalt/Metadaten unveränderlich (Trigger `fn_datev_exporte_schutz`) |
| Status | CHECK erweitert um `'ersetzt'` | Lebenszyklus: `erstellt → validiert/uebertragen/fehler/ersetzt` |

Schutz-Trigger (Migration): DELETE immer verboten; nach `eingefroren_am` sind nur noch
`status` (erlaubte Übergänge) und `fehler_details` änderbar; Einfrieren erfordert Hash
UND Inhalt gemeinsam; Hash/Inhalt/eingefroren_am sind nach dem Setzen unveränderlich.
DB-seitige Verifizierbarkeit: `encode(sha256(datei_inhalt),'hex') = inhalts_hash`.

### Neue Exporte: ab sofort bei Erzeugung eingefroren

`src/app/api/datev/export/route.ts` (POST): CSV **innerhalb** der Transaktion erzeugen
(Reihenfolge umstellen: Belege lesen → `generateExtf` → INSERT `datev_exporte` inkl.
`datei_inhalt`, `inhalts_hash = sha256(csv)`, `eingefroren_am = now()`,
`datei_groesse_bytes = csv.length`, `erzeugtAm`-Header aus DEMSELBEN Zeitpunkt, der als
`created_at` gespeichert wird: `const erzeugt = new Date()` einmal erzeugen, in
`generateExtf` UND als expliziter `created_at`-Spaltenwert im INSERT verwenden).
Damit gilt: ausgelieferte Datei == gespeicherte Datei == jeder Re-Download, bitgleich.
Hash-Berechnung in Node: `createHash('sha256').update(csvBuffer).digest('hex')`.

### Re-Download: gespeicherte Bytes zuerst

`src/app/api/datev/export/[id]/route.ts` (GET): wenn `datei_inhalt IS NOT NULL` →
exakt diese Bytes ausliefern (Header wie bisher, Dateiname `datei_pfad`). Nur für
Alt-Exporte ohne Inhalt (einzig der 2024-Stapel, bis er eingefroren ist) bleibt der
bestehende Regenerationspfad. KEIN stilles Nachfrieren beim Download (Einfrieren ist
ein auditiertes Ereignis, kein Nebeneffekt).

### `friereExportEin(tx, exportId)` — Erstfassung einfrieren

Neue Datei `src/lib/datev-versionen.ts`. Ablauf (innerhalb der übergebenen
Transaktion, Aufrufer: Nacherfassungs-Commit BER-119; zusätzlich manuell über
`POST /api/datev/export/[id]/einfrieren` für den Operator):

1. Export sperrend lesen (`SELECT … FOR UPDATE`); ist `eingefroren_am` gesetzt →
   no-op, zurück.
2. **Vorbedingung Rekonstruierbarkeit:** `firmen.updated_at < export.created_at`
   für die Firma des Session-Mandanten — sonst Abbruch mit Fehler
   `"Firmenkonfiguration wurde nach dem Export geändert (<updated_at>) — Erstfassung nicht sicher rekonstruierbar. Manuell klären."`
   (Ist-Stand erfüllt die Bedingung: Firma 01 zuletzt 14.07., Export 20.07.)
3. Datei regenerieren EXAKT wie der heutige Re-Download (gleiche SELECT-Reihenfolge
   `ORDER BY beleg_datum, beleg_nr`, `erzeugtAm` aus `created_at`, Firmenwerte aus
   `firmen`, `meta.gegenkonto`-Fallback — die Beleg-Spalten `gegenkonto`/
   `bu_schluessel`/`dokument_fehlt` sind zu diesem Zeitpunkt noch NULL/false und
   erzeugen dieselben Bytes wie der Alt-Code; der Guard läuft VOR jedem Beleg-UPDATE).
4. `UPDATE datev_exporte SET datei_inhalt=…, inhalts_hash=…, eingefroren_am=now(),
   datei_groesse_bytes=… WHERE id=…`.
5. Audit (beleg_id NULL — per Migration erlaubt für Export-Aktionen):
   `aktion='export_eingefroren'`, `alter_wert=NULL`,
   `neuer_wert='<datei_pfad> sha256:<hash> — Erstfassung vor Nacherfassung eingefroren'`.

Bekannte, dokumentierte Einschränkung (in `docs/GOBD.md` vermerken): Der
Header-Zeitstempel der ORIGINAL heruntergeladenen Datei stammte aus dem
Erzeugungsmoment, die Regeneration nutzt `created_at` — die beim StB liegende Kopie
kann in Header-Feld 6 um Sekunden abweichen. Für alle NEUEN Exporte ist das behoben
(identischer Zeitpunkt, s. o.).

### `POST /api/datev/export/[id]/korrektur` — Korrekturfassung erzeugen

Request: `{ grund: string }` (Pflicht, nicht leer → sonst 422
`"Korrekturgrund ist Pflicht"`). `[id]` = beliebige Fassung; die Route arbeitet auf
der NEUESTEN Fassung der Wurzel (`COALESCE(wurzel_export_id, id)`).

Ablauf in `withMandant`-Transaktion:
1. Neueste Fassung sperren; ist sie nicht eingefroren → zuerst `friereExportEin`
   (deckt den Fall „Korrektur ohne vorherige Nacherfassung").
2. Belege der Wurzel laden (`WHERE datev_export_id = wurzelId`, Reihenfolge wie
   Export). Guard: existiert ein Beleg mit `zahlungsweg IS NULL` → 409
   `"Nacherfassung unvollständig: <beleg_nr> ohne Zahlungsweg"` (Schlüssel ist KEIN
   Guard — `vorsteuer_relevant=false`-Belege bleiben ohne).
3. Neue Datei generieren (aktueller Code: Beleg-Gegenkonto Spalte 8, Schlüssel
   Spalte 9, `dokument_fehlt`-Kennzeichen); `erzeugtAm = new Date()` == `created_at`
   der neuen Zeile.
4. INSERT neue Zeile: `version = alt.version + 1`,
   `wurzel_export_id = wurzelId`, `ersetzt_export_id = alt.id`,
   `korrektur_grund = grund`, Zeitraum/Jahr/Typ von der Wurzel übernommen,
   Summen NEU berechnet (Semantik unverändert: `COALESCE(gebucht_*, betrag_*)`),
   `anzahl_belege`, `datei_pfad = 'EXTF_Buchungsstapel_<jahr>_<label>_K<version>.csv'`
   (z. B. `EXTF_Buchungsstapel_2024_Jahr_K2.csv`), `datei_inhalt`, `inhalts_hash`,
   `eingefroren_am = now()`, `status='erstellt'`, `mandant_id`.
   **Belege werden NICHT angefasst** — `datev_export_id` bleibt auf der Wurzel
   (Whitelist erlaubt ohnehin kein `Wert → Wert`).
5. Alte Fassung: `UPDATE datev_exporte SET status='ersetzt' WHERE id = alt.id`.
6. Audit: `aktion='export_ersetzt'`, `beleg_id=NULL`,
   `alter_wert='<alt.datei_pfad> v<alt.version> sha256:<alt.hash>'`,
   `neuer_wert='<neu.datei_pfad> v<neu.version> sha256:<neu.hash> — Grund: <grund>'`.
7. Response: die CSV als Download (wie POST /export), Header `X-Export-Id`,
   `X-Version`.

### Export-Seite (`/export`)

- Tabelle „Bisherige Exporte" erweitert um Spalten `Fassung` (`v1`, `v2 (Korrektur)`)
  und `Status`; ersetzte Fassungen ausgegraut, Download-Link bleibt (eingefrorene
  Bytes). Tooltip/Zeile 2 zeigt `sha256:<hash-gekürzt>` und den Korrekturgrund.
- Je neuester, eingefrorener Fassung mit vollständig nacherfassten Belegen: Button
  „Korrekturstapel erzeugen" (Dialog fragt den Grund ab; Vorbelegung
  `Nacherfassung Zahlungsweg/Steuerschlüssel nach StB-Rückmeldung vom 22.07.2026`).
- Beschreibungstext der Karte ersetzt: „Erneuter Download liefert die gespeicherte
  Datei bitgleich" statt der Regenerations-Formulierung.

## Akzeptanzkriterien (prüfbar)

- [ ] Neuer Export (POST) speichert `datei_inhalt`, `inhalts_hash`,
      `eingefroren_am`, `datei_groesse_bytes`; die HTTP-Antwortbytes sind identisch
      mit `datei_inhalt` (Test: Hash der Response == `inhalts_hash`), und
      `encode(sha256(datei_inhalt),'hex') = inhalts_hash` in SQL.
- [ ] Re-Download eines eingefrorenen Exports liefert `datei_inhalt` bitgleich.
      Prüfbar nachts ohne Prod-Daten: der Code-Pfad liest bei
      `datei_inhalt IS NOT NULL` weder `firmen` noch `belege` (Unit-Test der
      extrahierten Funktion + Review-Punkt im PR) — damit ist der heutige Fehler
      (Firmenkonfiguration verändert ausgelieferte Stapel rückwirkend)
      konstruktiv geschlossen; Hash-Gleichheit `sha256(bytes) == inhalts_hash`
      im Unit-Test.
- [ ] `friereExportEin`: zweiter Aufruf ist no-op; bei
      `firmen.updated_at > export.created_at` bricht er mit der definierten Meldung
      ab (Test in Rollback-Transaktion).
- [ ] Korrektur-Route: ohne `grund` → 422; mit unvollständiger Nacherfassung → 409
      mit `beleg_nr`; im Erfolgsfall existieren zwei Zeilen (v1 `ersetzt` +
      eingefroren, v2 `erstellt` + eingefroren, `ersetzt_export_id`/
      `wurzel_export_id` korrekt), Audit `export_ersetzt` vorhanden. (Vollzug am
      Echtbestand erst Runbook M5.)
- [ ] Zweite Korrektur derselben Fassung scheitert am partiellen UNIQUE-Index

- [ ] DELETE auf `datev_exporte` und UPDATE von `datei_pfad`/`inhalts_hash` nach dem
      Einfrieren schlagen fehl (Testanhang T12).
- [ ] `/export` zeigt Fassung, Status, Hash-Kürzel; ersetzte Fassung bleibt abrufbar.
- [ ] `docs/DATEV.md` (Re-Download-Abschnitt) und `docs/GOBD.md` (neuer Abschnitt
      „Export-Fassungen und Inhalts-Hash", inkl. Header-Zeitstempel-Einschränkung
      der Alt-Erstfassung) sind aktualisiert;
      `docs/verfahrensdoku/AENDERUNGEN-v1.1.md` beschreibt den Fassungs-Prozess.
