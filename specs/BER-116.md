# BER-116: Zahlungsweg am Beleg — dreiwertig, Gegenkonto festgeschrieben

> **Pre-Flight:** Lies `CLAUDE.md`, dann `docs/UEBERGABE.md`, dann
> `docs/audits/2026-07-23-strukturpruefung-ausbaustufe.md`.
> **Voraussetzung:** Die konsolidierte Migration
> `specs/migrations/20260723_stb_rueckmeldung_konsolidiert.sql` ist angewendet
> (Baulauf-Schritt S1). Diese Story enthält KEINE eigene Migration.
> **Nicht parallel** zu BER-117/118/119 bauen — gleiche Dateien.

## Kontext

Rückmeldung des Steuerberaters (22.07.2026): Beim Abgleich des Buchungsstapels mit dem
Kontoauszug fehlten Belege. Pro Buchungssatz muss erkennbar sein, ob er dem
Geschäftskonto zuzuordnen ist (Konto **1800**); alles ohne diese Kennzeichnung gehört
auf ein Alternativkonto (**1810**). Nachgereicht: **2100** für privat verauslagte
Belege (Privatkonto).

Heute ist das Gegenkonto eine Konstante pro Firma (`firmen.datev_gegenkonto`, Wert
`1800`) und wird in `src/lib/datev.ts` (`row[7] = meta.gegenkonto`) für jeden Beleg
identisch gesetzt. Bar oder privat bezahlte Belege behaupten damit im Stapel, sie seien
über die Bank gelaufen.

**Ersetzt den Entwurf vom 22.07.2026:** Der dort vorgesehene Boolean
`ueber_geschaeftskonto` kann die vom StB beschriebene Welt (drei Konten) nicht abbilden
(Validierungsbericht, Befund 3). Das Linear-Issue beschreibt noch die Boolean-Fassung;
**diese Spec ist maßgeblich.**

## Anforderung

Bei der Freigabe wird der Zahlungsweg als Pflichtauswahl ohne Vorbelegung erhoben,
dreiwertig, und das aufgelöste Gegenkonto am Beleg festgeschrieben; der Export nimmt
das Gegenkonto vom Beleg.

## Entschiedenes Design

### Datenmodell (bereits in der konsolidierten Migration enthalten — nicht neu anlegen)

| Spalte | Typ | Bedeutung |
|---|---|---|
| `belege.zahlungsweg` | `text`, CHECK `IN ('geschaeftskonto','alternativkonto','privat')`, NULL = noch nicht erhoben | die Auswahl |
| `belege.gegenkonto` | `varchar(10)`, CHECK `~ '^[0-9]{4,9}$'`, NULL = Altbestand | das bei der Freigabe aufgelöste DATEV-Gegenkonto |
| `firmen.datev_gegenkonto_alternativ` | `varchar(10) NOT NULL DEFAULT '1810'` | Konto für `alternativkonto` |
| `firmen.datev_gegenkonto_privat` | `varchar(10) NOT NULL DEFAULT '2100'` | Konto für `privat` |

`firmen.datev_gegenkonto` (`'1800'`) bleibt unverändert der Geschäftskonto-Fall.
Beide neuen Spalten sind ab Festschreibung Teil der Trigger-Whitelist mit
`NULL → Wert`-Ausnahme und Kopplung (nur gemeinsam änderbar) — siehe Migration.

**Warum beides speichern:** Das aufgelöste Konto am Beleg macht Export und Re-Export
unabhängig von späteren Änderungen der Firmenkonfiguration (heute verändert eine
Konfigänderung ausgelieferte Stapel rückwirkend — `docs/DATEV.md` behauptet
fälschlich das Gegenteil, Korrektur siehe unten). Der `zahlungsweg` bleibt als
fachliche Auswahl erhalten (Anzeige, spätere CAMT-Zuordnung).

### Auflösung (serverseitig, einzige Stelle)

In `src/app/api/belege/[id]/freigeben/route.ts`:

```
zahlungsweg = body.zahlungsweg            // 'geschaeftskonto' | 'alternativkonto' | 'privat'
gegenkonto  = CASE zahlungsweg
                WHEN 'geschaeftskonto' THEN firmen.datev_gegenkonto
                WHEN 'alternativkonto' THEN firmen.datev_gegenkonto_alternativ
                WHEN 'privat'          THEN firmen.datev_gegenkonto_privat
```

Die Firmen-Zeile wird wie in `src/app/api/datev/export/route.ts` über
`mandanten m JOIN firmen f ON f.firma_nr = m.firma_nr WHERE m.id = session.mandantId`
gelesen (via `sql`, vor der `withMandant`-Transaktion).

### Pflichtauswahl ohne Vorbelegung

- UI: Drei Radio-Buttons im Freigabe-Formular, **keiner vorselektiert**. Labels
  (Konten dynamisch aus der Firmen-Konfiguration):
  - `Geschäftskonto (1800)`
  - `Andere Karte/Konto (1810)` — Geschäftszahlungen über andere Karten/Konten,
    zur Abgrenzung (Betreiber-Klärung 23.07.2026)
  - `Privat verauslagt (2100)`
- Freigabe-Button bleibt `disabled`, solange keine Option gewählt ist.
- Server: fehlt `body.zahlungsweg` oder ist der Wert nicht einer der drei Literale →
  **HTTP 422**, `{ "error": "Zahlungsweg ist Pflicht: Geschäftskonto, anderes Konto oder privat verauslagt" }`.
  Die Serverprüfung gilt unabhängig vom UI-Zustand (kein Vertrauen in den Client).
- Bewusst kein stiller Default: ein vergessener Klick erzeugt exakt den gemeldeten
  Abgleichsfehler.

### Persistenz und Audit (in der bestehenden `withMandant`-Transaktion der Freigabe)

1. `UPDATE belege SET zahlungsweg = $1, gegenkonto = $2 WHERE id = $3`
   — **vor** dem Status-UPDATE auf `geprueft` (der Beleg ist noch offen, kein
   Trigger-Konflikt).
2. Audit-Eintrag (Aktion existiert nach Migration im CHECK):
   `INSERT INTO audit_log (beleg_id, mandant_id, aktion, alter_wert, neuer_wert)
   VALUES (id, session.mandantId, 'zahlungsweg_gesetzt', NULL,
   '<zahlungsweg> → <gegenkonto>')` — z. B. `geschaeftskonto → 1800`.

### Export

`src/lib/datev.ts`:
- `DatevBeleg` um `gegenkonto?: string | null` und (für BER-117 vorbereitet)
  `bu_schluessel?: string | null` erweitern.
- `belegRow`: `row[7] = b.gegenkonto ?? meta.gegenkonto;` — Fallback auf die
  Firmenkonstante NUR für Altbestand ohne Wert (bis BER-119 nacherfasst hat).
  `meta.gegenkonto` bleibt im Typ `DatevMeta` erhalten.
- Beide Export-Routen (`src/app/api/datev/export/route.ts`,
  `src/app/api/datev/export/[id]/route.ts`): `gegenkonto` (und `bu_schluessel`,
  `dokument_fehlt` — BER-117/118) in die SELECT-Spaltenlisten aufnehmen.

### Anzeige

- Detailansicht (`src/app/belege/[id]/page.tsx`), Abschnitt „Belegdaten": neue Zeile
  `Zahlungsweg` mit Wert `Geschäftskonto (1800)` / `Andere Karte/Konto (1810)` /
  `Privat verauslagt (2100)` / `—` (NULL). Anzeige aus `zahlungsweg` + `gegenkonto`
  des Belegs (nicht aus der Firmen-Konfiguration).
- Belegliste (`src/app/belege/page.tsx`): neue Spalte `Zahlung` zwischen `Konto` und
  `Kanal`, Kürzel `GK` / `AK` / `PR` / `—` mit `title`-Attribut in Langform.
  (Der Filter „Dokument fehlt" kommt mit BER-118; hier kein Filter.)

Label-Mapping als Konstante in `src/lib/format.ts`:
`export const ZAHLUNGSWEG_LABELS: Record<string, { kurz: string; lang: string }>`
mit den drei Schlüsseln (`geschaeftskonto` → `GK`/`Geschäftskonto`, `alternativkonto`
→ `AK`/`Andere Karte/Konto`, `privat` → `PR`/`Privat verauslagt`).

  Alternative, bewusst nicht gewählt — dokumentiert im Spalten-Kommentar).
