# BER-113: Beleg manuell erfassen + Originaldokument nachreichen

> **Pre-Flight:** Lies `CLAUDE.md`, dann `docs/UEBERGABE.md`.

## Kontext

Belege müssen erfassbar sein, bevor das Originaldokument vorliegt (Eigenbeleg).
Das Dokument wird später nachgereicht; die Zuordnung muss zuverlässig gelingen.

## Entschiedenes Design

- **Nachreichen per Upload direkt am Beleg** (Detailansicht) — explizite Zuordnung,
  damit es keinen Abgleich gibt, der misslingen kann.
- **Kennzeichnung:** Status `klaerungsbedarf` + `stb_vermerk` „Eigenbeleg —
  Originaldokument fehlt noch", nach dem Upload auf „nachgereicht am …" aktualisiert.
  Geht über die DATEV-Zusatzinformation an den StB (BER-109).
- **Freigabe blockiert**, solange keine Seite am Beleg hängt.

## Verifizierte Randbedingungen

| Fakt | Konsequenz |
|---|---|
| `eingangskanal` kennt bereits `frontend_upload` | kein Constraint-Umbau |
| `beleg_seiten` UPDATE per Trigger **generell gesperrt** | Seiten nicht umhängbar → Upload muss am Zielbeleg erfolgen |
| `belege.gobd_hash` steht auf der Festschreibungs-Sperrliste | Nachreichen **nur vor der Freigabe** → daher die Freigabe-Sperre |
| Duplikat-Index nur `WHERE gobd_hash IS NOT NULL` | Belege ohne Dokument kollidieren nicht |
| Edge-Action `archive-beleg-seite` liefert `storagePath` + SHA-256 | Upload-Route kann sie wiederverwenden |
| `naechste_beleg_nr(uuid)` → json | Nummernvergabe wie beim automatischen Eingang |

## Akzeptanzkriterien

- [x] `dashboard_service` kann Belege und Seiten anlegen (Grants **und** RLS-Policies)
- [x] GoBD-Felder bleiben beim INSERT unerreichbar (`geprueft_am` etc.)
- [x] Policies mandantengebunden und auf Entwurfsstatus beschränkt
- [ ] `POST /api/belege` — manueller Beleg mit Eigenbeleg-Vermerk
- [ ] `POST /api/belege/[id]/dokument` — Upload → Edge → `beleg_seiten` + Hash + Vermerk
- [ ] Freigabe-Route: 422 ohne Dokument
- [ ] UI: „Neuer Beleg"-Formular, Upload-Bereich in der Detailansicht

## Stand

**DB-Schicht fertig und auf Prod angewandt** (Migrationen `20260719222122`,
`20260719222201`). Befund dabei: `dashboard_service` hatte weder INSERT-Rechte noch
INSERT-Policies — die einzige vorhandene Policy verlangt ein Admin-JWT. Grants allein
hätten nicht gereicht.

**Offen: die App-Schicht** (zwei Routen + UI). Bewusst nicht im selben Zug gebaut, weil
der Upload-Pfad GoBD-kritisch ist (Hash, Storage, Duplikatschutz) und sich ohne
Passkey-Login nicht durchklicken lässt — die End-to-End-Prüfung muss vom Operator kommen.
