# BER-114: Zeitanzeige in Europe/Berlin statt Laufzeit-Zeitzone

> **Pre-Flight:** Lies `CLAUDE.md`, dann `docs/UEBERGABE.md`.

## Kontext

Alle Uhrzeiten im Dashboard lagen 2 Stunden hinter MESZ.

## Ursache

`getHours()`/`getDate()`/`getMonth()` liefern die Zeit in der Zeitzone der
**Laufzeit**. Die Detailansicht ist eine Server Component und wird auf Vercel
gerendert — dort läuft die Runtime in **UTC**.

Nicht nur kosmetisch:

- **Audit-Log**: GoBD-Protokolleinträge mit falscher Uhrzeit
- **Datumssprung**: Zeitstempel zwischen 00:00 und 02:00 MESZ liegen in UTC noch am
  Vortag — das angezeigte Datum wäre falsch gewesen

## Akzeptanzkriterien

- [x] `datum()`/`datumZeit()` formatieren explizit in `Europe/Berlin` (MEZ/MESZ automatisch)
- [x] Anzeigeformat unverändert (`D.MM.YYYY`, `HH:MM`)
- [x] `ttmm()` in `datev.ts` ebenfalls zeitzonenfest
- [x] Verifiziert mit erzwungener Laufzeit-Zeitzone

## Technische Notizen

Formatierung über `Intl.DateTimeFormat(..., { timeZone: "Europe/Berlin" })` mit
`formatToParts`, damit das bestehende Format (Tag ohne führende Null, Monat
zweistellig) erhalten bleibt.

`ttmm()` war auf UTC zufällig korrekt — ein reines `DATE` kommt als UTC-Mitternacht
an. Bei negativem UTC-Offset wäre das DATEV-Belegdatum jedoch um einen Tag verschoben
gewesen; das wäre ein echter Buchungsfehler und nicht nur ein Anzeigeproblem.

**Verifikation** (Laufzeit per `TZ` erzwungen):

| Eingabe | Ausgabe |
|---|---|
| `21:50Z` Sommer | `23:50` |
| `22:30Z` am 19.07. | `20.07. 00:30` |
| `23:30Z` am 15.01. | `16.01. 00:30` |
| DATEV TTMM `2024-01-30` | `3001`, auch unter `TZ=America/New_York` |
